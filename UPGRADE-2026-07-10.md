# Upgrade @aiiq/mcp for the 2026-07-09/10 API changes

Handoff spec, written 2026-07-10 from the site-repo session that shipped the API changes
(aiiq PRs #236 and #237, merged and **live in production**). Delete this file once shipped.

## Why (what changed in the API)

Two breaking changes are live at aiiq.org:

1. **`GET /api/rankings` is now a catalog.** Entries are
   `{ id, rankingName, rankingType, dimension, direction, modelCount, url }` — **no inline
   `models` arrays**. Full leaderboards moved to **`GET /api/rankings/:id`** (standard envelope
   `apiVersion/methodologyVersion/updatedAt` + the old inline ranking shape with `models`).
2. **Dimension slugs renamed** in every response: `app-building` → `frontend-engineering`,
   `production-engineering` → `backend-engineering` (model `dimensions` maps, ranking ids
   `frontend-engineering-iq`/`backend-engineering-iq`, benchmark `dimension` fields).

Additive changes worth adopting:

3. **`/api/v1/` is the canonical, pinnable base** (permanent alias of `/api/`). Configured
   clients — this package — should use it.
4. **`GET /api/domains`** + **`GET /api/domains/:slug`** — applied-capability domains
   (`cybersecurity`, `bio`, `machine-learning`): per-model domain composite IQs
   (`models: [{ model, compositeIQ, benchmarksCovered, benchmarksTotal, modelUrl }]`) plus
   benchmark leaderboards (`results: [{ model, harness, score, modelUrl }]`). The site's
   remote MCP server now has 9 tools (added `list_domains`, `get_domain`).
5. `GET /api/openapi.json` — OpenAPI 3.1 spec (source of truth for every shape below).
   Human docs moved to `https://www.aiiq.org/docs/api/`. `GET /api/` returns a JSON
   discovery index. All endpoints now send CORS `*` and explicit `Cache-Control`.

## Current package state (verified against this repo, v0.1.1)

- `src/client.ts:22` — `DEFAULT_BASE = 'https://www.aiiq.org'`; tools pass paths like
  `'/api/rankings'`.
- `src/tools/rankings.ts:19` `listRankings` — fetches `/api/rankings`, maps a catalog
  client-side. **Still works** but now redundant: the response already is a catalog
  (with extra `modelCount` + `url` fields the mapping drops).
- `src/tools/rankings.ts:31` `getRanking` — fetches `/api/rankings`, `find(r => r.id === id)`.
  **Broken**: returns a catalog entry with no `models`.

## Required changes

1. **Base URL:** point the client at the canonical versioned base. Either set
   `DEFAULT_BASE = 'https://www.aiiq.org/api/v1'` and change tool paths to drop the `/api`
   prefix (`/rankings`, `/models`, …), or keep the path style and make the base
   `https://www.aiiq.org` + rewrite paths to `/api/v1/...`. Pick whichever is cleaner in
   this codebase; the requirement is that every request hits `/api/v1/*`. Keep the
   `AIIQ_API_BASE` env override semantics intact (document what a custom base must now
   look like).
2. **`getRanking`:** fetch `GET /rankings/{id}` directly. On 404 (`{ error, code:
   'not_found' }`), fetch the catalog and return the available-ids error message as today.
3. **`listRankings`:** pass the catalog payload through (keep or extend the field mapping —
   consider including `modelCount`; it's useful to agents).
4. **New tools for parity with the remote server (9 total):**
   - `list_domains` → `GET /domains` (id, name, benchmarkCount, url per domain)
   - `get_domain` → `GET /domains/{slug}`; 404 → error listing available domain slugs
5. **Slug hygiene:** grep the repo (source, tests, README, tool descriptions, examples) for
   `app-building` and `production-engineering`; replace with the new slugs.
6. **Types:** update/add response types (catalog entry, ranking detail, domain summary,
   domain detail). `https://www.aiiq.org/api/openapi.json` is the authoritative schema —
   fetch it rather than guessing.
7. **Tests:** update fixtures for the catalog/detail split; add coverage for the two domain
   tools and the getRanking 404 path.

- Note for the release description: requires nothing from users; fixes `get_ranking`
  against the 2026-07-10 API and adds domain tools.

## Context that prevents wrong turns

- Do NOT add header-based version negotiation; the API versions by path only.
- The API compatibility policy (see `https://www.aiiq.org/docs/api/`): additive changes
  anytime; the `/api/v1/` contract never removes/renames fields. Pinning to v1 is the
  point of this change — future site-side breaking changes will ship at `/api/v2/`
  without breaking this package again.
- The site repo's remote MCP implementation (`lib/mcp-tools.js` in shea256/aiiq) is the
  behavioral reference for all 9 tools if anything here is ambiguous.
